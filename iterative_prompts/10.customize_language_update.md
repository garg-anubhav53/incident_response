# Prompt 10: Customer-Specific Outreach

## Intent

The status page communication is a broadcast — one message for all customers. But for key accounts, a CSM or account team member needs to send a tailored email that addresses that customer's specific situation. This is not primarily about tone. It is about **selecting the right details for that customer**: which of their specific products were affected, whether their particular integrations were impacted, what their history with Abnormal means for what they need to hear, and what questions they are most likely to ask.

A customer who has been with Abnormal for three years and runs high email volume through IES needs to hear something very different from a new customer who only uses the Portal. A customer whose SOAR integration depends on remediation status needs explicit confirmation of remediation health — information that might be buried or absent in the public status page. A volume user of AISM during an ATO incident needs confirmation that account takeover monitoring was unaffected, even if the public update doesn't call that out.

The extraction pipeline already knows what's broken, what's not, what caused it, how severe it was, and which security functions were impacted. This feature adds a generation layer on top of that shared understanding, producing a customer-specific email from a natural language description of the customer. The model selects which details from the extraction to surface, which to omit, and how to frame them for that specific relationship.

This transforms the tool from a status page writer into an incident communications platform — the status page is the public broadcast, and customer-specific outreach is the targeted follow-up that surfaces the details each account actually needs.

## Where it lives in the app

The results page currently shows the status page communications at the top (the Abnormal-statuspage-style display with the staged updates). This feature adds a new section directly below the status page output, visually separated but clearly part of the same analysis flow.

The flow should feel natural: the IC or CSM sees the public communication first, then can immediately generate a private, tailored version for specific customers without re-running any analysis. The same extraction data powers both.

## What the user does

The user sees a section titled something like "Customer-Specific Outreach." There is a text input area where they describe the customer in natural language. Examples of what someone might type:

- "Long-time enterprise customer, 5+ years, high IES volume, their CISO reads incident comms directly — give full root cause and timeline detail"
- "They're a heavy AISM user, make sure to confirm that account takeover monitoring was not affected"
- "Their security team runs SOAR playbooks off our remediation API — they need to know if automated response was impacted"
- "New customer, non-technical buyer, they just need to know their email is protected, keep it simple and reassuring"
- "This customer had a rough experience with our last outage and renewal is coming up — be transparent and acknowledge the impact directly"
- "FedRAMP customer, formal register required, their compliance team will review this"
- "They primarily use IES and EPR — only mention products that are relevant to them"

After entering the description, the user clicks a generate button. The system produces a draft email — subject line and body — calibrated to that customer context.

The user should be able to generate multiple outreach emails for different customers within the same incident, without any of them interfering with each other. Each generation is independent and uses the same underlying extraction data.

## Technical Requirements

### Generation

This requires a third Anthropic API call (separate from extraction and status page generation). The call should receive:

1. The full structured extraction output (same data that powers the status page communications)
2. The already-generated status page communications (so the model knows what was said publicly and can be consistent)
3. The natural language customer description
4. The security function classification (detection_status, remediation_status) so the email can speak precisely about whether that customer's security coverage was affected

The system prompt for this call should instruct the model to write an email (subject line and body) that:

- Is consistent with the public status page communication in facts and timeline, but may include more or fewer details depending on the customer context
- Adjusts technical depth based on the customer description — a CISO gets root cause mechanism detail, a non-technical buyer gets plain language reassurance
- Addresses the specific products or services the customer uses if mentioned in the description, mapping them against the extraction's affected and unaffected service lists
- Matches the emotional register to the relationship context — a customer with a recent bad experience gets more transparency and empathy, a routine notification gets professional brevity
- Never contradicts the public status page communication — the email can say more, but it cannot say different things about what happened or what's affected
- Follows the same rules about internal detail exclusion (no engineer names, no infra hostnames, no PR numbers) — the customer email is still an external communication
- Includes a clear call to action appropriate to the customer: "no action needed" for most, "you may want to verify your SOAR integration" for API-dependent customers, "please reach out to your CSM with any questions" for high-touch accounts
- Is formatted as a real email someone could send — greeting, body paragraphs, sign-off. Not a status page update pasted into an email

### UI

The section should include:

- A text area for the customer description (freeform, no structured fields — the whole point is natural language flexibility)
- A generate button
- The output displayed as a formatted email preview — subject line visible separately, body rendered cleanly
- A copy button that copies the subject and body as formatted text
- The ability to generate additional emails for other customers without clearing previous ones — each generation should appear as a separate card or block, so the CSM can prepare several outreach emails during one session

Think about the visual hierarchy. The status page communications are the primary output and should remain visually dominant. The customer outreach section is a secondary tool that some users will use and others will skip. It should be present but not compete for attention with the main output.

### Error handling

If the customer description is empty and the user clicks generate, show a gentle prompt asking them to describe the customer. If the API call fails, show the error without disrupting the existing results on the page. If the extraction data is missing fields that would help personalize the email (e.g., no security function classification), the generation should still work — it just produces a more generic email and notes what it couldn't personalize.

### Consistency safeguard

The most important constraint is factual consistency with the public communication. The email must not contradict the status page. It can elaborate on root cause, add reassurance about unaffected services, adjust tone, or include customer-specific context — but the facts (what happened, when, what's affected, what's not) must match. The system prompt should make this explicit.

A secondary consistency concern: if the user generates emails for multiple customers from the same incident, the facts should be identical across all of them. Only the tone, depth, and framing should change. This is naturally handled by using the same extraction data for every generation, but the system prompt should reinforce it.

## What this is NOT

This is not a CRM integration. It does not send emails. It does not look up customer data from any system. It does not know anything about the customer beyond what the user types into the text box. It is a generation tool that produces draft text for a human to review, possibly edit, and send through their own email client.

This is also not a template system. There are no predefined customer tiers or communication templates. The natural language input is the entire configuration layer. This keeps it flexible and avoids the failure mode of rigid templates that don't fit real customer relationships.
