# Prompt 6: Security Function Impact Classification

Add a feature that classifies whether an incident impacts detection capabilities, remediation capabilities, or other functionality. This classification is distinct from the pattern classification in Prompt 5 (which identifies *how* the incident happened — deployment, infra cascade, etc.). This one identifies *what security function is impacted* — which directly determines communication urgency and customer risk exposure.

This matters because Abnormal is a security product. A portal outage is inconvenient. A detection failure means threats are getting through. Those require fundamentally different urgency in communications, and misclassifying one as the other is the highest-stakes error this tool can make.

---

## The Three Categories

**Detection** — The system's ability to identify threats is degraded or offline. This includes: inbound email scanning (IES), behavioral analysis, threat scoring, AI phishing detection (AIPC), account takeover monitoring (ATO), and misdirected email prevention. If any of these are impacted, customers may be exposed to threats without knowing it. This is the highest-urgency category.

**Remediation** — The system identifies threats correctly but cannot act on them. This includes: email quarantine/deletion, automated response actions, SOAR API execution, and any handoff to external systems like Microsoft Graph for email removal. Threats are detected and visible, but automated response is delayed or broken. High urgency, but less than detection — customers can still see threats and act manually.

**Other** — Everything that isn't detection or remediation. This includes: portal access, authentication, SIEM API delivery, notification delivery, email productivity features (EPR), customer support portal, and platform/infrastructure issues that don't affect the security pipeline. These are operational inconveniences, not security exposure. Lower urgency in communications.

### Why this is hard to get right

The difficulty is that the raw incident data uses internal service names (like `api-gateway`, `portal-auth-service`, `remediation-service`), not product names. The LLM needs to map from internal service names to security function categories — and it needs to do this even when the service name is ambiguous. For example, `api-gateway` could serve detection API calls, remediation API calls, or portal requests. The classification has to come from the *symptoms and context* (what's failing, what errors say, what engineers discuss), not just the service name.

The second difficulty is partial overlap. An infrastructure cascade might take down both detection and remediation simultaneously. The classification needs to handle multi-category impact, not force a single label.

---

## Extraction Prompt Changes

Add a new field to the JSON output schema in `EXTRACTION_SYSTEM_PROMPT`, after `pattern_classification`:

**`security_function_impact`** — an object with these fields:

- `primary_category`: one of `detection`, `remediation`, `other`
- `secondary_category`: one of `detection`, `remediation`, `other`, or `null` if only one category is affected
- `confidence`: high, medium, or low
- `reasoning`: 2-3 sentences explaining the classification, specifically citing what evidence connects the failing service to a security function
- `detection_status`: one of `fully_operational`, `degraded`, `offline`, `unknown` — always populated, even if detection isn't the primary impact. This field exists so that the generation step always knows whether to include "detection remains fully operational" or not.
- `remediation_status`: same options as above, same rule — always populated.

Add a new rule to the CRITICAL RULES section:

Rule 13 — SECURITY FUNCTION CLASSIFICATION: Classify the incident's impact on security functions using these signals:

For detection impact, look for: errors in email scanning or processing pipelines, IES service failures, threat scoring or ML model errors, behavioral analysis failures, references to missed threats or false negative spikes, ATO monitoring failures, AIPC errors. Key CloudWatch message patterns include phrases referencing email processing, threat detection, scanning pipeline, model inference, or scoring.

For remediation impact, look for: failures in automated response actions, email quarantine or deletion errors, Microsoft Graph API failures, SOAR API errors, references to "remediation" or "response" in service names or error messages, engineers discussing threats that were detected but couldn't be acted on.

For other, look for: portal or UI errors with no mention of detection or remediation impact, authentication failures that block visibility but not protection, SIEM delivery failures (outbound integration only), notification delays, EPR features, platform or infrastructure issues where engineers confirm detection and remediation are unaffected.

Critical safeguards for this classification:
- If the service name is generic (like `api-gateway` or `platform`), do NOT classify based on service name alone. Look at what the errors describe, what the Slack thread discusses, and what metrics are degraded to determine which security function is actually impacted.
- If engineers explicitly state in the Slack thread that detection is unaffected ("detection still running," "emails are being processed," "security is not impacted"), trust that and classify as `other` — then set `detection_status` to `fully_operational`.
- If you cannot determine whether detection is impacted from the available data, set `detection_status` to `unknown` and flag this in `data_gaps`. Do NOT default to `fully_operational` when you're uncertain — that would give customers false assurance.
- If the incident impacts infrastructure that serves multiple functions (e.g., a shared database or API gateway), classify based on the most severe function impacted and use `secondary_category` for the other. In this case, set both detection and remediation status fields based on available evidence.
- When in doubt, classify UP in severity (detection > remediation > other). A false alarm about detection impact is better than missing a real detection gap.

## Generation Prompt Changes

Add this section to `GENERATION_SYSTEM_PROMPT`, after the required fields specification:

URGENCY CALIBRATION BY SECURITY FUNCTION:

The `security_function_impact` field from the extraction determines the tone and urgency of your communications. Apply these adjustments:

If detection is impacted (`detection_status` is `degraded` or `offline`): This is the highest urgency. The title should reference threat detection or email security directly. Every update must prominently state what security coverage is affected. Do NOT use minimizing language ("minor issue," "brief disruption"). Do use direct language ("threat detection is experiencing delays," "email security scanning is temporarily degraded"). The "what's not affected" field should focus on whether remediation still works for already-detected threats.

If remediation is impacted but detection is operational: Moderate-high urgency. The title should reference response actions or remediation. Every update must explicitly state that detection remains fully operational and threats are being identified — the gap is in automated response only. Guide customers: "Detected threats will be visible in the portal and can be acted on manually."

If neither detection nor remediation is impacted (other): Standard urgency. The title references the specific operational issue (portal access, API performance, notification delays). The most important sentence in every update is the confirmation that detection and remediation remain fully operational. Use this exact framing when the data supports it: "[Detection/email security] and [remediation/automated response] services remain fully operational and continue to protect against threats."

Always populate the "what's not affected" required field using the `detection_status` and `remediation_status` values from the extraction. If both are `fully_operational`, say so explicitly. If either is `unknown`, hedge: "We are confirming the status of [detection/remediation] services and will update shortly."

## UI Changes

### Status badge enhancement

Currently, the status badge shows Resolved/Degraded/Outage based on severity. Add a security function indicator next to it.

If detection is impacted, show an additional red badge: "Detection Impact" — this should be visually prominent and impossible to miss.

If remediation is impacted but detection is fine, show an orange badge: "Remediation Impact."

If neither is impacted, show a small green badge: "Security Functions Operational."

Display these badges in the same `overall-status` div, to the right of or below the existing severity badge.

### Detection/remediation status in the extraction expander

In the existing "Structured Incident Analysis" expander, add a new section at the top (before Timeline) titled "Security Function Status." Show detection status and remediation status side by side, each with a colored indicator: green circle for `fully_operational`, yellow for `degraded`, red for `offline`, gray for `unknown`. Show the classification reasoning below.

### Validation integration

Update the `validate_communications()` function to add one more check: if `detection_status` is `fully_operational` (confirmed), verify that the word "detection" or "email security" or "threat protection" appears in the communication with positive framing ("remains operational," "continue to protect," "fully operational"). If detection is confirmed fine but the communication doesn't say so, flag it as a high-severity warning — this is the single most important piece of reassurance for customers of a security product.

Similarly, if `detection_status` is `degraded` or `offline`, verify that the communication does NOT contain false reassurance about detection. If it says "detection remains fully operational" when detection is degraded, flag that as a critical error.

## Connection Points

This feature integrates with all three existing Prompt 5 features:

**Pattern matching (Feature 1):** The pattern classification tells you HOW the incident happened. The security function classification tells you WHAT is impacted. They're complementary — a `deployment_induced` pattern could impact detection, remediation, or other depending on what was deployed. Both should appear in the inference log.

**Deployment correlation (Feature 2):** The deployment correlation flags candidates. The security function classification tells you whether that deployment affected a security-critical function. If a HIGH-relevance deployment is flagged AND the security function impact is detection, that's the most urgent combination — the UI should make this very visible.

**Required fields (Feature 3):** The "what's not affected" required field is now informed by `detection_status` and `remediation_status`. The generation prompt should use these fields to populate the "what's not affected" content precisely rather than relying on the LLM to infer it.

## Testing

Run with the original 5 test files and verify:

- `primary_category` = `other` (the test incident is an API performance issue, not a detection or remediation failure)
- `detection_status` = `unknown` (the test data doesn't confirm detection status one way or the other — there's no IES metric or Slack confirmation)
- `remediation_status` = `unknown` (same reason)
- Both `unknown` values appear in `data_gaps`
- The UI shows the gray "unknown" indicator for both detection and remediation status, not a false green
- Communications use hedging language around security functions since status is unconfirmed
- The security function badge shows appropriately (not "Security Functions Operational" — that would be wrong since we don't know)

This last point is critical and is the main accuracy trap: the test incident has `api-gateway` as the affected service, which sounds like it could impact detection. But the data doesn't confirm whether detection traffic flows through this gateway. The correct classification is `other` with `unknown` detection/remediation status — not `detection` (overreaction) and not `other` with `fully_operational` (false assurance). The system must tolerate ambiguity here.
