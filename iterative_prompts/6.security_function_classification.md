# Prompt 6: Security Function Impact Classification

Add a feature that classifies whether an incident impacts detection capabilities, remediation capabilities, or other functionality. This classification is distinct from the pattern classification in Prompt 5 (which identifies *how* the incident happened — deployment, infra cascade, etc.). This one identifies *what security function is impacted* — which directly determines communication urgency and customer risk exposure.

This matters because Abnormal is a security product. A portal outage is inconvenient. A detection failure means threats are getting through. Those require fundamentally different urgency in communications, and misclassifying one as the other is the highest-stakes error this tool can make.

---

## The Three Categories

**Detection** — The system's ability to identify threats is degraded or offline. This includes: inbound email scanning (IES), behavioral analysis, threat scoring, AI phishing detection (AIPC), account takeover monitoring (ATO), and misdirected email prevention. If any of these are impacted, customers may be exposed to threats without knowing it. This is the highest-urgency category.

**Remediation** — The system identifies threats correctly but cannot act on them. This includes: email quarantine/deletion, automated response actions, SOAR API execution, and any handoff to external systems like Microsoft Graph for email removal. Threats are detected and visible, but automated response is delayed or broken. High urgency, but less than detection — customers can still see threats and act manually.

**Other** — Everything that isn't detection or remediation. This includes: portal access, authentication, SIEM API delivery, notification delivery, email productivity features (EPR), customer support portal, and platform/infrastructure issues that don't affect the security pipeline. These are operational inconveniences, not security exposure. Lower urgency in communications.

### Why this is hard to get right

The difficulty is that raw incident data uses internal service names (like `api-gateway`, `portal-auth-service`, `remediation-service`), not product names. The LLM needs to map from internal service names to security function categories — and it needs to do this even when the service name is ambiguous. For example, `api-gateway` could serve detection API calls, remediation API calls, or portal requests. Classification must come from the *symptoms and context*, not just the service name.

The second difficulty is partial overlap. An infrastructure cascade might take down both detection and remediation simultaneously. The classification must handle multi-category impact, not force a single label.

---

## Extraction Prompt Changes

**`security_function_impact`** added to JSON output schema:

- `primary_category`: one of `detection`, `remediation`, `other`
- `secondary_category`: one of `detection`, `remediation`, `other`, or `null` if only one category is affected
- `confidence`: high, medium, or low
- `reasoning`: 2-3 sentences citing what evidence connects the failing service to a security function
- `detection_status`: one of `fully_operational`, `degraded`, `offline`, `unknown` — always populated
- `remediation_status`: same options — always populated

**Rule 13 — SECURITY FUNCTION CLASSIFICATION** added to CRITICAL RULES.

Critical safeguards (now formalized):
- Service name alone never determines classification — classify from symptoms and engineer discussion
- Engineers confirming detection unaffected → `detection_status=fully_operational`
- If uncertain whether detection is impacted → `detection_status=unknown`, add to data_gaps. Never default to `fully_operational` when uncertain
- Ambiguous infrastructure → classify UP in severity (detection > remediation > other)
- **New cross-check constraint**: `portal_auth_failure` pattern is explicitly incompatible with `detection_status=degraded/offline`. If this combination appears, it is flagged by `validate_extraction_consistency()` as a high-severity consistency error, since portal issues do not impair email detection.

## Generation Prompt Changes

URGENCY CALIBRATION section in `GENERATION_SYSTEM_PROMPT`:

**detection degraded/offline**: Highest urgency. Title references threat detection or email security. Every update states what security coverage is affected. No minimizing language.

**remediation degraded, detection operational**: Moderate-high urgency. Title references response/remediation. Every update explicitly states detection is fully operational. Include: "Detected threats are visible in the portal and can be acted on manually."

**Both operational (primary_category=other)**: Standard urgency. The "what's not affected" field (second sentence in every update) reads: "Email security detection and automated remediation remain fully operational and continue to protect against threats."

**detection_status unknown**: Do NOT write "detection remains operational." Write: "We are confirming the status of detection services and will provide an update shortly."

---

## Validation Changes

`validate_communications()` checks (applied per stage):

**Check 1 — detection confirmation**: If `detection_status=fully_operational`, every communication must mention detection/email security. If not present, this is a high-severity warning — confirmed operational detection is the single most important reassurance for a security product's customers.

**Check 2 — false reassurance guard**: If `detection_status=degraded/offline` and the communication contains phrases like "detection remains fully operational" or "detection is fully operational," flag as CRITICAL ERROR. Publishing false reassurance about detection status when detection is actually impaired is the worst possible outcome.

`validate_extraction_consistency()` adds:

**Consistency check**: `portal_auth_failure` + `detection_status` in (`degraded`, `offline`) → high-severity consistency flag: "Portal/UI failures do not affect email detection. Review security function classification." Displayed in the Quality Checks zone before communication validation results.

## UI Changes

### Status badge enhancement

Security function badge added to the right of the severity badge in the status page header:
- Detection degraded/offline → red badge: "Detection Impact"
- Remediation degraded/offline, detection fine → orange badge: "Remediation Impact"
- Both fully_operational → green badge: "Security Functions Operational"
- Either unknown → gray badge: "Security Status Unconfirmed"

### Detection/remediation status in structured analysis

"Security Function Status" section appears at the top of the "Structured Incident Analysis" expander (before Timeline). Shows detection and remediation status side-by-side with colored pills (green/yellow/red/gray). Classification reasoning shown as a caption below.

### Quality Checks zone — consistency flags

`validate_extraction_consistency()` results appear in the Quality Checks zone above communication validation. High-severity flags shown as `st.error()`, medium as `st.warning()`. These catch logical contradictions between pattern classification and security function impact before the IC reads the communications.

## Connection Points

**Pattern matching (Prompt 5 Feature 1):** Pattern = how it happened. SFI = what is impacted. They're complementary — a `deployment_induced` pattern could impact detection, remediation, or other. Both appear in the inference log. Their cross-consistency is validated by `validate_extraction_consistency()`.

**Deployment correlation (Prompt 5 Feature 2):** If a HIGH-relevance deployment is flagged AND SFI primary_category is detection, that is the highest-urgency combination. The Quality Checks zone surfaces this prominently.

**Required fields (Prompt 5 Feature 3):** "What's not affected" is now directly populated from `detection_status` and `remediation_status` values — the generation prompt specifies exact language per status value rather than letting the LLM infer it.

## Testing

Run with the original 5 test files and verify:

- `primary_category` = `other` (the test incident is an API performance issue, not a detection or remediation failure)
- `detection_status` = `unknown` (the test data doesn't confirm detection status — there's no IES metric or Slack confirmation)
- `remediation_status` = `unknown` (same reason)
- Both `unknown` values appear in `data_gaps`
- UI shows gray "Security Status Unconfirmed" badge — NOT the green "Security Functions Operational" badge
- Communications use hedging language ("We are confirming the status of detection services...")
- Consistency check: `portal_auth_failure` + `unknown` detection status does NOT trigger a consistency flag — `unknown` ≠ `degraded/offline`, so the combination is valid
- If test data were modified to show `detection_status=degraded`, then a communications message saying "detection remains fully operational" WOULD trigger the critical false-reassurance check

The main accuracy trap in this test: `api-gateway` sounds like it could impact detection. The correct classification is `other` with `unknown` detection/remediation — not `detection` (overreaction) and not `fully_operational` (false assurance). The system must tolerate ambiguity here. Rule 0 (null over hallucination) and the `unknown` status option exist precisely for this case.
