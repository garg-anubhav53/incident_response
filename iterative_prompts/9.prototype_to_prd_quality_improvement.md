# Prototype-to-PRD Alignment Audit

You are auditing a Streamlit application (`app.py`) that generates incident communications for Abnormal AI. The PRD makes specific claims about what the system does. Your job is to verify that the actual implementation delivers on those claims, identify gaps, and fix them. The PRD will be submitted alongside this prototype, so any claim that isn't backed by working code is a credibility problem.

Read the full app.py first. Then work through each section below. For each item, determine: does the code actually do this? If partially, what's missing? If it's there but fragile (works for the sample data but might not generalize), flag that too.

---

## 1. THE MOST IMPORTANT CLAIM: Security Function Impact Drives Communication Content

The PRD says: "Every incident should be clearly labeled: is this a detection issue, a remediation issue, an operational issue, or a mixed situation? That classification should flow directly into the generated communication."

**Verify:**
- Does the extraction prompt actually produce `detection_status` and `remediation_status` as independent fields?
- Does the generation prompt read those fields and produce meaningfully different language based on them?
- If detection is "fully_operational" — does the generated text include something like "threat detection and remediation remained fully operational" in the second sentence?
- If detection is "degraded" or "offline" — does the generated text change to reflect that customers may be exposed?
- Is this difference testable? Could a reviewer change the detection_status in the extraction JSON and see the communication change accordingly?

**If there's a gap:** The generation prompt needs explicit conditional logic: "If detection_status is fully_operational, the second sentence must state that detection and remediation are unaffected. If detection_status is degraded or offline, the second sentence must state the specific impact to protection and customer action guidance must include defensive measures."

---

## 2. "WHAT'S NOT AFFECTED" AS SECOND SENTENCE

The PRD says this is "enforced structurally, never buried or trimmed."

**Verify:**
- In the generation system prompt, is there an explicit instruction that the second sentence of every stage's communication must address what's not affected?
- Is there a trim priority rule that says if word count needs to be reduced, "what's not affected" is never the thing that gets cut?
- Look at the actual generated output for the sample incident. Is "what's not affected" actually the second sentence in each of the four stages (Investigating, Identified, Monitoring, Resolved)?
- If it's not consistently second, is it at least always present and prominent?

**If there's a gap:** Add explicit structural instruction to the generation prompt: "Structure every stage message as: (1) What is happening/happened. (2) What is NOT affected — this sentence is mandatory and must never be removed for brevity. (3) Customer impact. (4) What we are doing. (5) Customer action. (6) Next update timing."

---

## 3. FIVE FIXED VERIFICATION QUESTIONS WITH COPY-LOCK

The PRD says: "Non-negotiable checklist with verbatim evidence. Copy-to-publish is locked until all five are addressed."

**Verify:**
- What are the five questions? Are they hardcoded or generated?
- Does each question display verbatim evidence (actual log lines, actual metric values, actual Slack messages) rather than AI summaries?
- Is there actually a copy-to-clipboard or export function?
- Is that function actually disabled/locked until all five items are checked?
- If an item is marked "disputed," does it actually block the copy action?
- Is the verification state stored in session_state so it survives Streamlit reruns?

**If there's a gap:** The copy-lock is a specific, visible, testable claim. If the button exists but isn't actually locked, or if the lock logic has edge cases (e.g., page refresh resets verification state), fix it. This is something a reviewer might specifically test.

---

## 4. CONSISTENCY VALIDATION: FALSE REASSURANCE DETECTION

The PRD calls this "the highest-priority automated check."

**Verify:**
- Does `validate_extraction_consistency()` actually exist and run?
- Does it check: if detection_status is "degraded" or "offline" AND the generated communication contains language like "detection remains fully operational" or "all security functions are operating normally" → flag as error?
- Does it check the specific contradiction: portal_auth_failure pattern + detection degraded/offline?
- Are the results of this validation actually displayed in Zone 3 (Quality Checks)?
- What does it look like when a check fails? Is it visually distinct (red error vs amber warning)?

**If there's a gap:** This is the most critical safety check in the entire system. If it exists but only catches exact string matches (and the LLM uses slightly different phrasing), it's not reliable. Consider whether the check should also run through the LLM as a validation pass, or whether it needs a broader set of string patterns.

---

## 5. DEPLOYMENT CORRELATION: PRE-LLM, DETERMINISTIC

The PRD says: "Multi-metric consensus onset detection. Provides an independent cross-check against the AI's analysis."

**Verify:**
- Does `correlate_deployments()` actually run before the extraction LLM call?
- Does it actually require ≥2 metrics anomalous in the same 5-minute window for onset detection?
- Does it actually separate deployments into pre-onset (potential causes) and post-onset (potential worsening factors)?
- Are PR numbers actually excluded from the displayed output?
- Is the output actually displayed in Zone 1 before the status page draft?
- If the deterministic correlation contradicts the LLM's classification, is that contradiction actually visible to the user? Or do they appear in different zones with no explicit connection?

**If there's a gap:** The "independent cross-check" claim is only meaningful if contradictions are surfaced. If the deployment correlation says "no suspicious deployments found" but the LLM says "deployment_induced," the user should see that disconnect called out, not just infer it by reading two different sections.

---

## 6. RULE 0: NULL OVER HALLUCINATION

The PRD says: "Unknown fields must be empty, never invented."

**Verify:**
- Is Rule 0 actually in the extraction system prompt?
- Does it apply to all fields or just some?
- Look at the actual extraction output for the sample data. Are there any fields that look like they might be invented rather than extracted? Common culprits:
  - `affected_regions` — does the data actually specify regions, or is the AI guessing?
  - `customer_impact_summary` — is this derived from evidence or generated from the incident pattern?
  - `data_gaps` — does the AI actually populate this, or is it empty? (An empty data_gaps array is suspicious — almost every incident has something the data doesn't confirm.)

**If there's a gap:** Test by removing a data source (e.g., remove the Slack thread file) and re-running. Do fields that depended on Slack data become null/empty, or does the AI fill them in from guesswork?

---

## 7. RESULTS PAGE: SIX ZONES IN DECISION ORDER

The PRD says: "Pre-Flight Context → Status Page Preview → Quality Checks → IC Verification → Analyst View → Raw Extraction Data"

**Verify:**
- Are these actually rendered in this order in the code?
- Are they actually labeled with the zone names and subtitles described in the codebase walkthrough?
- Are Zones 5 and 6 actually collapsed by default?
- Is the visual hierarchy correct — can someone make a publish/no-publish decision from Zones 1-3 without scrolling through technical details?

**If there's a gap:** Zone ordering is straightforward to fix. The more subtle issue is whether Zones 1-3 actually contain enough information for a publish decision, or whether critical context is buried in Zone 4 or 5.

---

## 8. FOUR-STAGE COMMUNICATIONS MATCHING ABNORMAL'S PATTERNS

The PRD says the output should match Abnormal's real status page patterns.

**Verify:**
- Does each stage communication include all six required elements? (What happened, what's not affected, customer impact, what we're doing, customer action, next update timing)
- Does the "customer action" element actually appear in the Investigating stage? The PRD claims it's required in ALL stages including Investigating (e.g., "No customer action is required at this time").
- Is stage ordering enforced in Python (not relying on LLM output order)?
- Does the Resolved stage include total incident duration?
- Does the Identified stage language actually tie to root_cause.status (confirmed vs hypothesized vs unknown)?
- Compare the generated output against a real Abnormal status update from status.abnormalsecurity.com. Does the format, tone, and structure feel like the same voice?

**If there's a gap:** The most likely gap is the customer action requirement in Investigating stage and the root_cause.status linkage in Identified stage. These are specific prompt-level rules that may or may not be present.

---

## 9. INFERENCE LOG PERSISTENCE

The PRD says: "Inference logs are already persisted to disk, so historical pattern analysis is a query over existing data."

**Verify:**
- Does `save_inference_log()` actually write to disk?
- What format? Where?
- Does it include the evidence_trace alongside the inference log?
- Does `load_inference_logs()` actually read prior logs?
- Are prior logs actually displayed anywhere in the UI?
- If I run the pipeline twice on the same data, do I get two log entries or does it overwrite?

**If there's a gap:** The persistence claim supports the "cross-incident pattern detection" feature in the path forward section. If logs aren't actually persisting or aren't structured enough to query across incidents, that future feature claim is hollow.

---


## 10. OVERALL UX COHERENCE

- On a fresh page load, is it obvious what to do?
- Could a reviewer who has never seen this tool before understand the results page in 60 seconds?

The last point is critical. The PRD describes a system where an IC can make a publish decision from the top of the results page. If the actual results page is so dense or confusing that a first-time user spends 2 minutes figuring out what they're looking at, the claim doesn't hold.

---
## PRIORITY ORDER FOR FIXES

If you find gaps, fix them in this order:

1. **False reassurance detection** — if this doesn't work reliably, the entire safety story collapses
2. **"What's not affected" as second sentence** — the single most repeated claim in the PRD
3. **Copy-lock on verification** — a specific, testable, impressive claim that the reviewer might try
4. **Security function classification driving communication content** — the core domain insight
5. **Zone ordering and labeling** — the information hierarchy claim
6. **Everything else** — word count badges, inference log persistence, deployment correlation details

For each fix, keep it minimal. Don't restructure the application. Add or modify the specific code that addresses the gap. Test with the sample data after each change.
