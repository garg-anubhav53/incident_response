# Pipeline Improvement Prompt

You are modifying app.py in a Streamlit incident communications tool. Make these thematic changes to improve accuracy, validation, and user experience.

---

## 1. EXTRACTION ACCURACY OVERHAUL

**Intent:** Prevent the most critical failure modes - confident wrong answers and security product false assurances.

**Changes:**
- Add Rule 0: "NULL OVER HALLUCINATION - Unknown fields must be null/[], never invented"
- Update Rule 2: "ROLLBACK = CONFIRMATION - Successful rollback confirms root cause even without engineer statement"
- Update Rule 11: "portal_auth_failure incompatible with detection_status=degraded/offline - portal issues don't impair email detection"
- Update Rule 14: "what_is_not_broken requires specific evidence (logged absence or engineer statement), not just service names that 'seem fine'"
- Add confidence structures note explaining three distinct confidence outputs for different consumers

**Why:** These rules address the highest-stakes accuracy issues for a security vendor - false confidence about detection status and invented facts that sound plausible.

---

## 2. GENERATION STRUCTURE & SECURITY FOCUS

**Intent:** Ensure communications follow Abnormal's actual status page patterns and never mislead about security coverage.

**Changes:**
- Make "what's not affected" the SECOND sentence in every update (never buried)
- Require customer action in ALL stages including Investigating ("No action is required at this time")
- Add word count trim priority: never cut the "what's not affected" sentence
- Tie Identified stage language to root_cause.status (confirmed/hypothesized/unknown)
- Add DST handling (PDT/PST conversion, always label "PT")
- Add concrete 79-word example message to anchor format

**Why:** Security customers need immediate reassurance about detection status. The second sentence placement mirrors Abnormal's real communications pattern.

---

## 3. VERIFIABLE EVIDENCE INFRASTRUCTURE

**Intent:** Enable the LLM to cite specific, verbatim evidence rather than summaries for the evidence trace system.

**Changes:**
- summarize_metrics(): Return two-part output - human summary + RAW ANOMALOUS VALUES with verbatim timestamp-value pairs
- build_user_message(): Internal type detection + annotate section headers with original filenames
- Update Rule 14 guidance: LLM must cite from RAW ANOMALOUS VALUES section using format "metric_name: value at timestamp"

**Why:** Evidence trace requires verbatim citations, not summaries. Raw metric values enable specific citations like "http_request_duration_seconds: 2.5 at 2025-01-15T14:20Z".

---

## 4. DEPLOYMENT CORRELATION INTELLIGENCE

**Intent:** Make deployment correlation more robust by requiring consensus and surfacing worsening factors.

**Changes:**
- Multi-metric consensus for onset: Require ≥2 metrics anomalous in same 5-minute window
- Add 30-minute post-onset window for deployments that may worsen (not cause) incidents
- Label post-onset deployments as "POST-ONSET" and sort separately
- Exclude PR numbers from returned flags (internal details)

**Why:** Single anomalous metrics can be noise. Post-onset deployments reveal incident evolution patterns.

---

## 5. COMPREHENSIVE VALIDATION SYSTEM

**Intent:** Catch both structural omissions and logical contradictions before publication.

**Changes:**
- validate_communications(): Remove Investigating exemption for customer action requirement
- validate_communications(): Add word count >110 warning
- Add validate_extraction_consistency(): Cross-check pattern vs security function impact
- SFI validation: Ensure detection status mentioned when confirmed, prevent false reassurance

**Why:** Multiple validation layers catch different problem types - structural gaps vs semantic contradictions.

## 6. UI ORGANIZATION & VISUAL HIERARCHY

**Intent:** Structure the results page for clear decision-making flow from most to least critical information.

**Changes:**
- 6 named zones in order: Pre-Flight Context → Status Page Preview → Quality Checks → IC Verification → Analyst View → Raw Extraction Data
- render_deployment_alerts(): Use .deploy-high CSS class, separate POST-ONSET expander

- render_evidence_trace(): Conclusion preview in headers, disputed cards auto-expand
- render_pattern_analysis(): Group inference entries by type with confidence summary

**Why:** ICs need to make publish/no-publish decisions quickly. Critical information appears first.
