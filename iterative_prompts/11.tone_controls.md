# Prompt 11: Adjustable Tone and Content Controls with Fast Regeneration

## Intent

The initial status page draft is generated by the extraction and generation pipeline using Sonnet — optimized for accuracy and completeness. But once that draft exists, the IC or comms reviewer often needs to adjust it: more technical, more reassuring, shorter, more detail about timeline, less detail about root cause. These adjustments don't require re-analyzing the incident. They're rewriting the same facts with different emphasis.

This is a rewrite task, not an analysis task. That distinction is the entire architectural insight. Rewrites can use a faster, cheaper model because the hard reasoning (what happened, what caused it, what's affected) is already done and locked in. The rewrite model receives the structured extraction as settled facts and only needs to rephrase and restructure — something a fast model does well.

The result is near-instant regeneration when the IC adjusts a slider. No loading spinner, no waiting. The communication updates as fast as the IC can think about what they want.

## Where it lives

On the results page, directly below the status page communications output and above the customer-specific outreach section. Visually, it should feel like controls attached to the status page output — not a separate feature, but a refinement layer on the same content.

## What the user sees and does

A set of controls that adjust the generated communications. When any control changes, the status page communications above regenerate immediately using the fast model. The extraction data does not change — only the prose output.

Three controls:

**Tone** — a selector or slider between three positions: "Technical," "Balanced," and "Reassuring." Technical means more precise language, mentions of service names, and factual density appropriate for a sophisticated SOC team. Reassuring means empathetic, plain-language, focused on impact and resolution rather than cause, appropriate for a broad customer base that wants to know "am I safe." Balanced is the default and matches the current generation output.

**Detail Level** — a selector between "Brief," "Standard," and "Detailed." Brief means each status update is 2-3 sentences maximum — headline facts only. Standard is the current default (the 3-5 sentence range from the generation prompt). Detailed means each update can be longer, include more timeline precision, and reference specific affected capabilities.

**Emphasis** — a selector for what the communication should lead with: "Customer Impact," "Resolution Progress," or "Root Cause." This shifts what gets the most prominent position in each update. Customer Impact leads with what the customer is experiencing. Resolution Progress leads with what's being done about it and when it will be fixed. Root Cause leads with why it happened (appropriate for postmortem-style communications or when customers are asking "how did this happen").

When any control changes, the status page output above should regenerate. The regeneration should be fast enough that it feels interactive — under 2 seconds in most cases.

## Architecture: Smart Model Selection

This is the key technical decision, and it should be documented clearly in any code walkthrough or comments.

The app uses two different models for two different jobs:

The **extraction pipeline** (Prompt 2) and **initial generation** (Prompt 3) use `claude-sonnet-4-20250514`. These require strong reasoning — analyzing raw data, cross-referencing sources, making judgment calls about severity and scope. Quality matters more than speed here, and these run once per incident.

The **tone/detail/emphasis regeneration** uses `claude-haiku-3-5-20241022`. This is a rewrite task on settled facts. The extraction output is passed as immutable context, and the model only needs to rephrase and restructure. Haiku is significantly faster and cheaper, and more than capable of prose reformulation when the reasoning is already done.

This is not a compromise — it's the correct model for the task. Document this decision in the codebase with a comment explaining the rationale: "Haiku for rewrites because the hard reasoning (extraction) is already complete. The rewrite model receives structured facts and only adjusts tone, length, and emphasis. This enables near-instant regeneration when controls change."

## How regeneration works

When a control changes, the app makes a new API call using Haiku. The call sends:

1. The full structured extraction output (unchanged — this is the source of truth for all facts)
2. The current control settings (tone, detail level, emphasis)
3. The security function classification (detection_status, remediation_status) so the rewrite preserves accurate security function language

The system prompt for the rewrite call should be a streamlined version of the generation prompt. It should include:

- The same rules about internal detail exclusion, timestamp format (PT), and status stage structure (Investigating/Identified/Monitoring/Resolved)
- The same required fields specification (all six fields must still appear in every update)
- The same urgency calibration based on security function impact
- Modified instructions for tone, detail, and emphasis based on the current control settings
- An explicit instruction: "The extraction data below is the single source of truth. Do not alter any facts. Do not change what's affected, what's not affected, the timeline, the severity, or the root cause. Only change how these facts are expressed — the language, the length, the emphasis, and the level of technical detail."

The output format is the same as the original generation — the same JSON structure with staged communications. The render function should be able to display it identically.

## Preserving the original

When controls change and a rewrite is generated, the original Sonnet-generated communications should be preserved in session state. Include a small "Reset to original" link or button near the controls that restores the initial output. This lets the IC experiment with different settings without fear of losing the default version.

## Interaction with other features

**Required fields validation (Prompt 5 Feature 3):** The validation function should run on every rewrite, not just the initial generation. If a rewrite drops a required field (possible with the Brief detail level), the validation warning should appear immediately. This is the main quality risk of fast regeneration — the rewrite model might cut important content when asked to be brief.

**Evidence trace (Prompt 7):** The evidence trace does not change when tone controls are adjusted. It's tied to the extraction, not the generation. This is correct — the evidence for "what caused it" doesn't change because the IC asked for a more reassuring tone.

**Customer-specific outreach (Prompt 10):** The tone controls affect only the public status page communications. Customer-specific emails are generated independently and have their own calibration via the natural language customer description. The controls should not affect customer emails, and the UI should make this separation clear.

## Performance expectations

Haiku rewrite calls should complete in 1-2 seconds for typical status page communications (4 updates, each under 100 words). If the call takes longer than 3 seconds, something is wrong — likely the prompt is too long or the max_tokens is set too high. Keep max_tokens at 1024 for rewrites (the output is shorter than the initial generation since facts are settled and the model only needs to rephrase).

Document the observed latency difference between the initial Sonnet generation and the Haiku rewrites in the code walkthrough. This is a concrete, measurable demonstration of smart model selection — the kind of thing a reviewer at an AI-native company will appreciate.

## What this is NOT

This is not a free-text editing interface. The IC cannot type arbitrary changes to the communication. That's a different feature (and a more complex one). This is parametric control — the IC adjusts knobs and the system regenerates. The output is always a complete, coherent set of communications, not a partially edited draft.

This is also not a prompt playground. The controls map to specific, meaningful communication dimensions (tone, detail, emphasis) that correspond to real editorial decisions an IC makes. They are not arbitrary prompt modifiers.
