# Prompt 5: MVP Feature Bundle — Pattern Matching, Deployment Correlation, Required Fields

Add three features to the existing Streamlit app. Each one touches the extraction prompt, the generation prompt, and the UI — so this prompt specifies all the connection points. Read the full prompt before making any changes, since the features share wiring.

---

## Feature 1: Incident Pattern Matching with Inference Logging

### What it does

The extraction pipeline classifies each incident into a known pattern type AND logs a structured record of what it inferred and why. Over time, these logs accumulate into institutional knowledge — "the last 3 incidents were deployment-induced, maybe we need better deployment gates."

### Extraction prompt changes

Add two new fields to the JSON output schema in `EXTRACTION_SYSTEM_PROMPT`, after `source_attribution`:

**`pattern_classification`** — an object with three fields: `primary_pattern` (one of: `deployment_induced`, `infrastructure_cascade`, `regional_failure`, `portal_auth_failure`, `third_party_dependency`, `integration_api_failure`, `silent_degradation`, `isolated_environment`, `unknown`), `confidence` (high/medium/low), and `reasoning` (2-3 sentences explaining why this pattern was selected and what evidence supports it).

**`inference_log`** — an array of objects, each with: `claim` (what was inferred), `sources` (which files/data points support it), `inference_type` (one of: `direct_observation`, `cross_reference`, `absence_of_evidence`, `engineer_confirmation`), and `confidence` (high/medium/low).

Add two new rules to the CRITICAL RULES section:

Rule 11 — PATTERN CLASSIFICATION: Classify the incident into exactly one primary pattern based on evidence. Provide definitions matching the pattern names above (deployment_induced means a code/config deploy correlates with onset AND is confirmed; infrastructure_cascade means multiple services fail simultaneously with no deployment trigger; etc — use the same definitions from our earlier incident type analysis). If evidence is ambiguous between two patterns, pick the stronger one and note the alternative in reasoning.

Rule 12 — INFERENCE LOG: For every factual claim in the analysis, log the inference chain. Each entry should state what was concluded, what data was used, and how the conclusion was reached. The four inference types map to reliability levels: `direct_observation` means the fact is explicitly stated in one source; `cross_reference` means it was derived by correlating multiple sources; `absence_of_evidence` means it's a conclusion based on what's NOT in the data; `engineer_confirmation` means engineers explicitly agreed on it in the Slack thread.

### Persistence

Write a function that saves a structured inference log to disk after each successful extraction. Store logs as JSON files in an `inference_logs/` directory, named with a UTC timestamp. Each log should contain: analysis timestamp, incident summary, pattern classification, the full inference log array, severity, confidence scores, and data gaps.

Write a companion function that loads all past inference logs from that directory, sorted newest first, for display in the UI.

Call the save function immediately after the extraction API call succeeds.

### UI additions

Add an expander below the status page output titled "Pattern Analysis & Inference Log." It should show:

The detected pattern name (human-readable, with underscores replaced by spaces and title-cased), a confidence emoji (green/yellow/red circle), and the reasoning text.

Below that, the inference chain — each entry displayed with an icon for its type (📌 direct observation, 🔗 cross-reference, ❓ absence of evidence, ✅ engineer confirmation), the claim in bold, confidence level, and sources in a caption.

If there are past inference logs (more than just the current one), show a divider and a "Recent Incident Patterns" section listing the last 5, each showing the pattern name and a truncated summary.

## Feature 2: Deployment Correlation

### What it does

Before the LLM runs, a Python function scans `github_deployments.json` and flags any deployment that happened within 60 minutes before incident onset. This surfaces deployment candidates immediately — independent of LLM reasoning — as a safety net that catches what the LLM might miss.

### Pre-processing function

Write `correlate_deployments(file_contents: dict) -> list` that:

1. **Finds incident onset time** from available data. Try PagerDuty's `created_at` first for a reliable structured timestamp. Then check Prometheus metrics for an earlier signal — scan each metric series, compute a baseline from the first two values, and find the first value exceeding 2x baseline. Use whichever timestamp is earliest. If neither source is available, return an empty list.

2. **Scans deployments** within the 60-minute window before onset. For each deployment in that window, compute: how many minutes before onset it occurred, and whether it deployed to the same service named in the PagerDuty alert.

3. **Scores relevance**: `HIGH` if the deployment targets the same service as the alert, `LOW` if it's a different service. Sort results with HIGH first, then by proximity to onset.

4. **Returns** a list of flagged deployments, each containing: service, title, author, timestamp, minutes before onset, whether it matches the alert service, relevance score, PR number, and description.

### Integration with the extraction pipeline

Call `correlate_deployments()` after reading uploaded files but before calling `run_extraction()`. Store the results in session state.

Pass the deployment flags into `run_extraction()`. If flags exist, append them to the user message as a new section: `=== PRE-COMPUTED DEPLOYMENT CORRELATION ===` followed by the flags as JSON, with a note saying "Use this to inform your root cause analysis, but verify with other sources before confirming causation."

### UI

Show deployment correlation results ABOVE the status page output as alert boxes:

For HIGH relevance deployments, show a warning banner: the deployment title, target service, and how many minutes before onset it was deployed, with "(same service as alert)" noted.

For LOW relevance deployments, collapse them into an expander: "N other deployment(s) in time window (different service)" with each listed as a single line.

## Feature 3: Required Communication Fields

### What it does

Every generated status update MUST contain six fields. If any are missing, the UI shows a validation warning. This ensures communications are always complete — especially "what's not affected," which is the single most critical field for a security vendor.

### Generation prompt changes

Replace the STRUCTURE PER UPDATE section in `GENERATION_SYSTEM_PROMPT` with a REQUIRED FIELDS specification. Every stage's message must include all six of:

1. **Affected service** — name the product or capability that's impacted, in customer-facing terms ("API endpoints," "the Abnormal Portal," "email threat detection").

2. **What's not affected** — explicitly state what continues to work normally. For a security product, this is the most important sentence in the communication. If the extraction confirms unaffected functionality, state it clearly: "Email detection and remediation remain fully operational." If unaffected scope is unknown, say: "Other services are not reporting issues at this time." Never omit this field.

3. **Customer impact** — what the customer is experiencing in plain language ("may experience slower response times," "may see errors when accessing the portal").

4. **Timestamp** — when the issue started, in PT with approximate language: "Starting around 2:20 PM PT."

5. **Next update commitment** — when the customer can expect the next communication. For Investigating: "within 30 minutes." For Identified: "within [estimated time]." For Monitoring: "we will continue monitoring for [duration]." Not needed for Resolved.

6. **Customer action** — what (if anything) the customer should do. Most incidents: "No action is required." If a workaround exists, state it. For Resolved: "If you continue to experience issues, please contact our support team."

Add an instruction: if the extraction data has gaps around any of these 6 fields, use hedging language ("some customers may...") rather than omitting the field. A missing field is worse than an approximate one.

### Post-generation validation

Write `validate_communications(comms: dict) -> list` that checks each stage's message for keyword indicators of each required field. Define a keyword list per field — for example, "unaffected" checks for phrases like "not affected," "remain," "operating normally," "fully operational," "continue to," "not impacted," "not reporting issues." Apply common-sense exemptions: skip the next-update check for Resolved stage, and skip customer-action for Investigating (where it's often implicit).

Return a list of warnings, each noting the stage, the missing field name, and a severity (high for "unaffected" since it's the most critical, medium for others).

### UI

After rendering the status page, before the extraction expander:

If there are high-severity validation warnings, show each as a red error box naming the stage and the missing field, with a note that it's critical for security vendor communications.

If there are medium-severity warnings, collapse them into an expander with a count.

If everything passes, show a green success message: "All communications pass required field validation."

## Integration Checklist

The coding agent should verify all these connections work end-to-end:

1. `EXTRACTION_SYSTEM_PROMPT` updated with rules 11-12 and new output fields (`pattern_classification`, `inference_log`)
2. `save_inference_log()` called after successful extraction
3. `correlate_deployments()` called before extraction, result passed to `run_extraction()`
4. Deployment flags appended to LLM user message in `run_extraction()`
5. `GENERATION_SYSTEM_PROMPT` updated with the 6 required fields spec
6. `validate_communications()` runs after generation, warnings displayed in UI
7. Deployment correlation alert box renders above status page output
8. Pattern analysis expander renders below status page output
9. Inference history panel shows past analyses if any exist
10. All new fields handled gracefully if missing — the LLM may not always return them, so use `.get()` with defaults everywhere
