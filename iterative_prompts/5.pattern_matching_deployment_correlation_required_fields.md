# Prompt 5: MVP Feature Bundle — Pattern Matching, Deployment Correlation, Required Fields

Add three features to the existing Streamlit app. Each one touches the extraction prompt, the generation prompt, and the UI — so this prompt specifies all the connection points. Read the full prompt before making any changes, since the features share wiring.

---

## Feature 1: Incident Pattern Matching with Inference Logging

### What it does

The extraction pipeline classifies each incident into a known pattern type AND logs a structured record of what it inferred and why. Over time, these logs accumulate into institutional knowledge — "the last 3 incidents were deployment-induced, maybe we need better deployment gates."

### Extraction prompt changes

Add two new fields to the JSON output schema in `EXTRACTION_SYSTEM_PROMPT`:

**`pattern_classification`** — an object with three fields: `primary_pattern` (one of: `deployment_induced`, `infrastructure_cascade`, `regional_failure`, `portal_auth_failure`, `third_party_dependency`, `integration_api_failure`, `silent_degradation`, `isolated_environment`, `unknown`), `confidence` (high/medium/low), and `reasoning` (2-3 sentences explaining why this pattern was selected and what evidence supports it, including any alternative considered).

**`inference_log`** — an array of objects, each with: `claim` (what was inferred), `sources` (which files/data points support it), `inference_type` (one of: `direct_observation`, `cross_reference`, `absence_of_evidence`, `engineer_confirmation`), and `confidence` (high/medium/low).

Rules added:

**Rule 11 — PATTERN CLASSIFICATION**: Classify into exactly one primary pattern. If ambiguous, pick the stronger one and explicitly note the alternative in reasoning. Important constraint: `portal_auth_failure` is incompatible with `detection_status=degraded/offline` — portal issues do not impair email detection. Any such combination should be flagged in data_gaps.

**Rule 12 — INFERENCE LOG**: For every factual claim in the analysis, log the inference chain. The four inference types map to reliability levels: `direct_observation` = fact explicitly stated in one source; `cross_reference` = derived by correlating multiple sources; `absence_of_evidence` = conclusion from what's NOT in the data; `engineer_confirmation` = engineers explicitly agreed in Slack. This log is the audit trail — every factual claim must appear here.

### Persistence

`save_inference_log(extraction)` saves to `inference_logs/{UTC_timestamp}.json` after each successful extraction. The log includes: analysis_timestamp, incident_summary, severity, pattern_classification, security_function_impact, inference_log, **evidence_trace**, confidence_scores, data_gaps.

Note: `evidence_trace` is included in the persisted log (added in Prompt 7) to enable future incident replay without re-running the LLM.

`load_inference_logs(limit=5)` loads newest-first for display in the UI.

### UI additions

Pattern Analysis & Inference Log rendered in a collapsed expander under the "Analyst View" zone. Shows:

- Detected pattern name (title-cased), confidence emoji, reasoning
- Inference chain confidence summary: "N high · N medium · N low"
- Inference entries grouped by type into 4 collapsible sub-groups (engineer_confirmation auto-expanded, others collapsed):
  - ✅ Engineer Confirmations
  - 📌 Direct Observations
  - 🔗 Cross-References
  - ❓ Absence of Evidence
- Recent Incident Patterns (last 5 from disk, current incident filtered out)

## Feature 2: Deployment Correlation

### What it does

Before the LLM runs, `correlate_deployments()` scans `github_deployments.json` and flags deployments in a window around incident onset. This is a deterministic check independent of LLM reasoning that surfaces deployment candidates the LLM might miss.

### Pre-processing function

`correlate_deployments(file_contents: dict) -> list`:

1. **Finds incident onset time** from PagerDuty `created_at` first. Then checks Prometheus for an earlier signal using **multi-metric consensus**: collects the first-anomaly timestamp per metric, then finds the earliest window where ≥2 metrics are simultaneously anomalous within 5 minutes. This prevents a single transient spike from setting a false early onset.

2. **Pre-onset window**: 60 minutes before onset. **Post-onset window**: 30 minutes after onset (these surface deployments that may have worsened the incident, not caused it).

3. **Scores relevance**: `HIGH` = same service as alert, pre-onset. `LOW` = different service, pre-onset. `POST-ONSET` = any service, after onset. Sorts: HIGH → LOW → POST-ONSET; by proximity within each group.

4. **Returns** list of flagged deployments. Note: `pr_number` is intentionally excluded from returned flags (PR numbers are internal details per extraction Rule 7 and should not surface in the UI).

### Integration with the extraction pipeline

`correlate_deployments()` runs before `run_extraction()`. Flags are appended to the user message as `=== PRE-COMPUTED DEPLOYMENT CORRELATION ===` with a note to verify before confirming causation.

`build_user_message(file_contents, deployment_flags)` now handles type detection internally and annotates each section header with the original filename so the LLM can produce accurate `source_file` citations in `evidence_trace`.

### UI

HIGH relevance alerts render as custom `.deploy-high` HTML cards (orange-bordered, not `st.warning()`) with service, title, author, timestamp, and `timing_label` (e.g. "15min before onset").

LOW relevance: collapsed expander titled "N deployment(s) in 60-min pre-onset window (different service)."

POST-ONSET: separate collapsed expander titled "N deployment(s) after incident onset (potential worsening factors)."

All three appear in the "Pre-Flight Context" zone above the status page.

## Feature 3: Required Communication Fields

### What it does

Every generated status update must contain specific content. Post-generation validation checks for this content using keyword heuristics and flags gaps before publication.

### Generation prompt changes

REQUIRED FIELDS — every stage message, in order:

1. **Affected service** — customer-facing name
2. **What's not affected** — MUST be the second sentence in every update; never cut to meet word count
3. **Customer impact** — plain-language description
4. **Timestamp** — PT with approximate language; DST handled (March–October = PDT = UTC-7)
5. **Next update commitment** — omit for Resolved only
6. **Customer action** — ALL stages including Investigating ("No action is required at this time"); Resolved uses contact-support language

Word count target: under 100 words. Trim priority: (1) next-update timing, (2) extra impact detail, (3) timestamp precision. NEVER trim field 2.

Identified stage language tied to root_cause.status: "confirmed" → "we have identified [abstract cause]"; "hypothesized" → "we believe we have identified what appears to be [abstract cause]"; "unknown" → skip Identified stage entirely.

### Post-generation validation

`validate_communications(comms, extraction)` runs 5 keyword checks + 2 SFI checks + 1 word count check per stage:

1. What's not affected (high severity if missing)
2. Customer impact (medium)
3. Timestamp (medium)
4. Next update commitment (medium, skipped for Resolved)
5. Customer action (medium, **no exemptions** — all stages including Investigating must have this)
6. Word count >110: medium warning
7. SFI: detection confirmed operational → must mention detection (high severity if missing)
8. SFI: detection degraded/offline → must NOT falsely claim detection is operational (critical)

`validate_extraction_consistency(extraction)` cross-checks pattern vs SFI:
- `portal_auth_failure` + detection degraded/offline → high flag
- root_cause confirmed + confidence low → medium flag
- `deployment_induced` + root_cause unknown → medium flag
- `third_party_dependency` + no external service in trigger → medium flag

### UI — Quality Checks zone

Consistency flags shown first (high as `st.error()`, medium as `st.warning()`).

Communication validation: red `st.error()` per high warning, medium warnings in collapsed expander. Green success div if all pass. Quality Checks zone appears between Status Page Preview and IC Verification.

## Integration Checklist

1. `EXTRACTION_SYSTEM_PROMPT` updated with Rules 0-14 (Rule 0 = null over hallucination; Rules 11-12 = pattern + inference)
2. `save_inference_log()` called after successful extraction; includes `evidence_trace` in persisted data
3. `correlate_deployments()` called before extraction; result stored in session state and passed to `run_extraction()`
4. `build_user_message()` handles type detection internally; annotates headers with filenames
5. `GENERATION_SYSTEM_PROMPT` updated with 6 required fields, what's-not-affected placement rule, word count trim priority, example message
6. `validate_communications(comms, extraction)` runs after generation; no customer-action exemption for Investigating
7. `validate_extraction_consistency(extraction)` runs after validation; `consistency_flags` stored in session state
8. Stage order enforced in `run_generation()` after LLM returns
9. Results rendered in 6 named zones with visual zone separators
10. All new fields handled gracefully if missing — use `.get()` with defaults everywhere

## Testing

Run the app with the original 5 test files and verify:

- Pattern classification = `deployment_induced` with high confidence
- Deployment correlation flags HIGH relevance pre-onset deployment (same service), LOW relevance pre-onset deployment (different service) — PR numbers do NOT appear in the UI
- All 4 communication stages pass the required fields validation
- "What's not affected" appears as the second sentence in every communication
- "No action is required" or equivalent appears in the Investigating stage
- Inference log has entries grouped across all 4 type categories; confidence summary shows at least 3 high-confidence entries
- Inference log file saved to `inference_logs/` including `evidence_trace`
- Consistency flags section shows nothing (no contradictions in the sample data)
