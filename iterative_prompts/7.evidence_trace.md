# Prompt 7: Evidence Trace & Human-in-the-Loop Verification

Add a feature that shows the incident commander exactly which pieces of raw source data support each conclusion the system reached. The goal is not just attribution ("this came from CloudWatch") but actual evidence — the specific log entry, the specific metric values, the specific Slack message — so an engineer can verify correctness in under 60 seconds without leaving the app.

This is the trust layer. Everything else in the system is valuable only if the IC trusts it enough to publish the communications. An IC who can't quickly verify the system's reasoning will rewrite the communications from scratch anyway, defeating the purpose of the tool.

---

## What an engineer needs to verify before publishing

There are exactly five claims an IC needs to check. Everything else is secondary. The evidence trace should be organized around these five, in this order:

1. **What's broken** — Is the system correct about which service is affected and what customers are experiencing?
2. **What caused it** — Is the root cause identification correct? Is the causal chain (trigger → mechanism → symptom) actually supported?
3. **When it started and ended** — Are the timeline boundaries right? Did degradation really begin when the system says it did?
4. **What's not broken** — Is the "unaffected" claim actually supported, or is the system guessing? This is the most dangerous claim to get wrong for a security product.
5. **How bad is it** — Is the severity assessment (degraded vs partial outage vs full outage) calibrated correctly?

Each of these should show the raw evidence, not a summary. The IC should be able to look at the quoted data and form their own independent judgment, then compare it against the system's conclusion.

---

## Extraction Prompt Changes

Add a new field to the JSON output schema in `EXTRACTION_SYSTEM_PROMPT`, after `inference_log`:

**`evidence_trace`** — an array of exactly 5 objects, one per verification question. Each object has:

- `question`: One of the five verification questions above, as a short label: `what_is_broken`, `what_caused_it`, `when_did_it_happen`, `what_is_not_broken`, `how_bad_is_it`
- `conclusion`: The system's one-sentence answer to this question
- `evidence`: An array of evidence items, each containing:
  - `source_file`: Which file this came from (e.g., "cloudwatch_logs.json", "incident_context.txt")
  - `raw_excerpt`: The exact text, log entry, or data point from the source — verbatim, not paraphrased. For JSON sources, this should be the specific object or value. For the Slack thread, this should be the exact message with timestamp and author.
  - `relevance`: One sentence explaining why this evidence matters for the conclusion
- `counter_evidence`: An array of any evidence that weakens or contradicts the conclusion, using the same structure as `evidence`. If there is no counter-evidence, this should be an empty array — but the model should actively look for contradictory signals rather than only collecting supporting evidence.
- `verification_suggestion`: A specific action the IC can take to verify this conclusion independently (e.g., "Check the Prometheus dashboard for api-gateway p99 latency between 14:15 and 15:00 UTC", "Confirm with the deployment author whether the timeout change was expected to affect connection pooling")

Add this rule to the CRITICAL RULES section:

Rule 14 — EVIDENCE TRACE: For each of the five verification questions, cite specific raw evidence from the source data. Requirements:
- Raw excerpts must be verbatim. Do not paraphrase or summarize log entries, metric values, or Slack messages. Copy them exactly as they appear in the data.
- For metrics, cite the specific timestamp-value pairs that matter — the baseline values, the first anomalous value, the peak, and the recovery point. Do not say "metrics showed elevated latency." Say "http_request_duration_seconds p99 went from 0.15 at 14:15 UTC to 2.5 at 14:20 UTC to 15.0 at 14:25 UTC."
- For the "what's not broken" question, evidence of absence is valid but must be labeled as such. If no errors appear for a service, that's evidence it's unaffected — but weaker evidence than an engineer explicitly confirming it in Slack. State which type of evidence you have.
- Always look for counter-evidence. If you conclude the root cause is a deployment but there's a Slack message where someone suggests a different cause, include that as counter-evidence even if it was later dismissed. The IC should see that the alternative was considered.
- The verification suggestion should be something the IC can actually do in under 2 minutes — checking a dashboard, asking a specific person, or running a specific query. Do not suggest vague actions like "investigate further."

## UI Implementation

### Evidence trace panel

Add a new section in the main display area, between the status page communications output and the existing expanders. Title it "Verify Before Publishing" with a shield or checkmark icon.

Display each of the five verification questions as a collapsible card. The card header should show:

- The question in plain language (e.g., "What caused it?")
- The system's conclusion as a single line
- A colored confidence indicator from the extraction's confidence scores (green/yellow/red)
- A verification status toggle: three states — unverified (gray, default), verified (green, IC clicked to confirm), disputed (red, IC clicked to flag disagreement)

When the card is expanded, show:

**Supporting evidence** — each evidence item displayed as a block with a light background. The source file name in a small label at the top. The raw excerpt in a monospace font to visually distinguish it from the system's words. The relevance note in regular text below.

**Counter-evidence** (if any) — same format but with a different background color (light amber/yellow) so it visually stands out as "things that cut the other way."

**Verification suggestion** — displayed at the bottom of the card in italics, as a concrete next step.

### Verification status persistence

Store the IC's verification decisions (verified/disputed for each of the five questions) in session state. If the IC disputes a claim, show a text input where they can note what's wrong. This isn't persisted to disk — it's for the current session only.

Show a summary bar above the communications output: "3 of 5 claims verified, 1 disputed, 1 unverified." If any claim is disputed, show a warning that the communications should be reviewed before publishing.

### Pre-publish checklist

Below the evidence trace, add a "Ready to Publish?" section. This is a simple checklist that the IC marks off manually:

- "I have verified the affected service is correct"
- "I have verified the root cause (or confirmed it's marked as unconfirmed)"
- "I have verified the timeline boundaries"
- "I have verified the 'unaffected' claims are accurate"
- "I have reviewed the generated communications for accuracy"

All five must be checked before a "Copy Communications" button becomes active. The button copies the full set of staged communications to clipboard as formatted text.

This is deliberately friction. The tool should make verification fast, but it should never make publishing thoughtless. An IC who clicks through the checklist in 30 seconds has still engaged with each question, which is better than a one-click publish.

## Trust-Building Details

These are small additions that compound into a significantly more trustworthy experience:

**Show what data was NOT available.** Below the evidence trace, include a brief section: "Data not available for this analysis." List any of the 5 standard files that were not uploaded, and note what verification those files would have enabled. For example: "github_deployments.json was not uploaded — deployment correlation could not be performed. Root cause confidence is reduced." This sets expectations honestly and explains why confidence might be lower.

**Highlight when the system changed its mind.** If the inference log shows that an early hypothesis was replaced by a later one (e.g., the Slack thread shows someone suggesting cause A, then the team converging on cause B), call this out explicitly in the evidence trace for the root cause question. Show both the initial hypothesis and the correction. This demonstrates that the system tracks the evolution of understanding, not just the final answer.

**Flag stale timestamps.** If the incident data timestamps are more than 24 hours old, show a subtle note: "This analysis is based on incident data from [date]. If the incident is still active, re-upload current data for an updated analysis." This prevents an IC from accidentally publishing stale communications.

**Show the mapping from internal to external language.** In the evidence trace, wherever the system translated a technical term to customer-facing language (e.g., "connection pool exhaustion" became "API performance degradation"), show both terms side by side. The IC can verify that the translation is accurate and doesn't lose important nuance.

## Connection Points

**With Prompt 5 Feature 1 (inference logging):** The evidence trace is a more detailed, UI-friendly version of the inference log. They share the same underlying data but serve different purposes — the inference log is for persistence and pattern analysis over time, the evidence trace is for real-time IC verification. Both should be generated from the same extraction call.

**With Prompt 5 Feature 3 (required fields validation):** If the evidence trace shows counter-evidence for the "what's not broken" question, AND the communication validation shows the "unaffected" field is present, the IC should see both — the communication claims something is unaffected, but there's some evidence that cuts against that. This is the highest-value overlap.

**With Prompt 6 (security function classification):** The evidence trace for the "what's not broken" question should explicitly reference the detection_status and remediation_status values from the security function classification. If detection_status is `unknown`, the evidence trace should show this and explain why — which is more useful than just a yellow dot in the classification panel.

- Verification suggestions are specific and actionable, not generic
- The pre-publish checklist blocks the copy button until all items are checked
- The "Data not available" section is empty (since all 5 files are uploaded for the test case)
