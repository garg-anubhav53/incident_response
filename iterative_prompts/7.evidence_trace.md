# Prompt 7: Evidence Trace & Human-in-the-Loop Verification

Add a feature that shows the incident commander exactly which pieces of raw source data support each conclusion the system reached. The goal is not just attribution ("this came from CloudWatch") but actual evidence — the specific log entry, the specific metric values, the specific Slack message — so an engineer can verify correctness in under 60 seconds without leaving the app.

This is the trust layer. Everything else in the system is valuable only if the IC trusts it enough to publish the communications. An IC who can't quickly verify the system's reasoning will rewrite the communications from scratch anyway, defeating the purpose of the tool.

---

## What an engineer needs to verify before publishing

There are exactly five claims an IC needs to check. Everything else is secondary. The evidence trace should be organized around these five, in this order:

1. **What's broken** — Is the system correct about which service is affected and what customers are experiencing?
2. **What caused it** — Is the root cause identification correct? Is the causal chain (trigger → mechanism → symptom) actually supported?
3. **When it started and ended** — Are the timeline boundaries right? Did degradation really begin when the system says it did?
4. **What's not broken** — Is the "unaffected" claim actually supported, or is the system guessing? This is the most dangerous claim to get wrong for a security product.
5. **How bad is it** — Is the severity assessment (degraded vs partial outage vs full outage) calibrated correctly?

Each of these should show the raw evidence, not a summary. The IC should be able to look at the quoted data and form their own independent judgment, then compare it against the system's conclusion.

## Extraction Prompt Changes

**`evidence_trace`** added to JSON output schema — an array of exactly 5 objects:

- `question`: `what_is_broken`, `what_caused_it`, `when_did_it_happen`, `what_is_not_broken`, `how_bad_is_it`
- `conclusion`: one-sentence answer
- `evidence`: array of items, each with:
  - `source_file`: exact filename (matches the `(filename: X)` annotation in the section headers)
  - `raw_excerpt`: verbatim — exact log lines, exact Slack messages with author+timestamp, exact metric values as "metric_name: value at timestamp." Never paraphrased.
  - `relevance`: one sentence explaining why this evidence supports the conclusion
- `counter_evidence`: array using same structure. Must be populated with anything that weakens the conclusion — empty array only if truly nothing contradicts.
- `verification_suggestion`: specific action the IC can complete in under 2 minutes (named dashboard, specific person, specific query — never "investigate further")

**Rule 14 — EVIDENCE TRACE** added to CRITICAL RULES.

### What_is_not_broken specifics (Rule 14d)

Do NOT list service names that "seem fine." Cite a specific absence: "No errors appear for [service] in [filename] during the incident window" OR an explicit engineer statement. If neither exists, use a single evidence item:
- `raw_excerpt`: "No error data found for this service in any provided source"
- `relevance`: "Absence of evidence — weaker than positive confirmation"

Never claim a service is unaffected based solely on its name.

### Metrics evidence (Rules 5 and 14b)

The `summarize_metrics()` function now provides a two-part output:

```
METRIC SUMMARY:
metric_name: N anomalies — baseline=X, first anomaly=Y at T1, peak=Z at T2

RAW ANOMALOUS VALUES (cite these verbatim in evidence_trace):
{"metric_name": [{"timestamp": "T1", "value": Y}, ...]}
```

The LLM should cite values from the RAW ANOMALOUS VALUES section in `evidence_trace`, not the summary counts. For example: `"http_request_duration_seconds: 2.5 at 2025-01-15T14:20Z"` — not `"latency was elevated."` This requires the metrics pre-processing to pass verbatim anomalous values to the LLM, which the current implementation does.

## UI Implementation

### "IC Verification" zone

Evidence trace appears in the "Incident Commander Verification" zone — after Quality Checks, before the analyst view sections. This ordering means an IC sees the output, checks it passed QC, then verifies the underlying reasoning before deciding to publish.

### Card header design (scannable without expanding)

Each card header shows:
```
[status icon] **Question Label** [confidence emoji] — _conclusion preview (80 chars)_
```

This means an IC can validate 3-4 cards in 10 seconds by reading headers, without expanding. Cards with `verify_status == "disputed"` auto-expand so disputed items are immediately visible. Verified cards stay collapsed (green checkmark in header is sufficient).

### Card content when expanded

**Supporting evidence** — `source_file` as a label, `raw_excerpt` in `st.code()` (monospace, visually distinct from system text), `relevance` as caption.

**Counter-evidence** — amber-tinted background (`#fffbeb` / `border-left: 3px solid #f59e0b`), same structure. Visually distinct from supporting evidence.

**Verification suggestion** — italicized at bottom of card.

### Verification status

Three-state toggle buttons per card: ✅ Verified / ❌ Disputed / ⬜ Reset.

Summary bar above all cards shows: "N/5 verified · N disputed (red) · N unverified." If any disputed, shows warning that disputed claims should be resolved before publishing.

If a card is disputed, a text input appears for the IC to note what's wrong with the conclusion. Notes are persisted in `verify_notes` session state for the current session.

### Stale data warning

If the detection timestamp from `timeline.detection_time_utc` or `timeline.onset_time_utc` is more than 24 hours old, shows: "Analysis based on data from [date] ([N]h ago). If the incident is still active, re-upload current data."

### Data availability section

Below the evidence cards, shows which of the 5 standard file types were provided for analysis. For any missing type, shows what verification capability was reduced. All 5 present → green confirmation. This sets honest expectations about confidence.

### Internal → customer language mapping

Collapsed expander showing internal terms found in source data (from `internal_details_to_exclude`) alongside the customer-facing description (from `customer_impact.description`). IC can verify the translation is accurate.

### Pre-publish checklist

"Ready to Publish?" section with 5 manual checkboxes:
- Affected service is correct
- Root cause is correct (or confirmed as unconfirmed)
- Timeline boundaries are correct
- 'Unaffected' claims are accurate for this security product
- Generated communications reviewed for tone and accuracy

Copy button (shows formatted communications as `st.code()`) is gated: unlocks only when all 5 checked AND zero cards are in disputed status. Deliberately structured friction — the IC must engage with each question before publishing becomes possible.

## Connection Points

**With Prompt 5 Feature 1 (inference logging):** Evidence trace and inference log share the same extraction call but serve different consumers. Inference log = audit trail + trend analysis over time. Evidence trace = real-time IC verification. Both are persisted to `inference_logs/` (evidence_trace added to the saved dict).

**With Prompt 5 Feature 3 (required fields validation):** If the evidence trace shows counter-evidence for "what's not broken" AND the communication validation shows the "unaffected" field is present, the IC sees both: the communication asserts something is unaffected, but there's evidence that cuts against it. Highest-value overlap between features.

**With Prompt 6 (security function classification):** The "what's not broken" evidence card should surface `detection_status` and `remediation_status` values. If status is `unknown`, the evidence trace should show this and explain why — which is more actionable than just a gray dot in the classification panel.

## Testing

Run with the original 5 test files and verify:

- All 5 evidence trace cards are populated with at least one supporting evidence item
- Card headers show a conclusion preview without expanding (scannable top-to-bottom)
- The "what caused it" card cites specific deployment data and specific Prometheus values showing the latency spike
- The "what's not broken" card cites specific absences or engineer statements — it should NOT claim detection is fine based solely on service name; the evidence item should use the "No error data found" template with "Absence of evidence" label if no confirmatory data exists
- The "when did it happen" card cites specific metric timestamp-value pairs from the RAW ANOMALOUS VALUES section (e.g., "http_request_duration_seconds: 2.5 at 2025-01-15T14:20Z") — not just anomaly counts
- Counter-evidence is populated where relevant (e.g., a different deployment as an alternative root cause candidate)
- Verification suggestions are specific and actionable (named dashboard, specific person, specific query)
- Pre-publish checklist blocks the copy output until all 5 items checked
- If any card is marked disputed, the copy output remains blocked even if all checklist items are checked
- Stale data warning does NOT appear for fresh data (timestamps within last 24 hours)
- "Data not available" section is empty when all 5 files are uploaded
