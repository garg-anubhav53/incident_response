# Prompt 1: Hardcoded Pipeline Test

## Overview
Build a Python script `pipeline_v1.py` that processes incident data from multiple sources and generates a structured analysis using the Anthropic API.

## Data Sources
Read 5 incident data files from a `data/` directory:

- `incident_context.txt` (freeform Slack thread + engineer notes)
- `cloudwatch_logs.json` (structured application logs)
- `prometheus_metrics.json` (time-series system metrics)
- `pagerduty_incident.json` (alert lifecycle)
- `github_deployments.json` (deployment history with diffs)

## API Integration
Send all file contents in a single Anthropic API call (claude-sonnet-4-20250514) with a system prompt that instructs the model to extract a structured JSON incident analysis. Print the structured JSON response.

### API Setup
```python
import os
from dotenv import load_dotenv
from anthropic import Anthropic

# Load environment variables from .env file
load_dotenv()

# Initialize Anthropic client
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))
```

## System Prompt
You are an incident analysis engine. You receive raw technical data from multiple sources about a production incident. Extract a structured analysis as JSON with exactly these fields:

```json
{
  "incident_summary": "One sentence: what happened, to what, for how long",
  "affected_service": "Service name from PagerDuty/logs",
  "severity": "From PagerDuty",
  "timeline": {
    "trigger_time_utc": "When degradation began in metrics",
    "detection_time_utc": "When alert fired",
    "acknowledged_time_utc": "When engineer responded",
    "mitigation_time_utc": "When fix was deployed",
    "recovery_time_utc": "When metrics returned to normal",
    "resolved_time_utc": "When incident was formally closed"
  },
  "root_cause": {
    "summary": "One sentence confirmed root cause",
    "trigger": "What specific change/event caused it",
    "mechanism": "How the trigger led to customer impact"
  },
  "customer_impact": {
    "description": "What customers experienced in plain language",
    "severity_assessment": "degraded_performance | partial_outage | full_outage",
    "affected_functionality": ["list of affected features/capabilities"]
  },
  "resolution": {
    "action_taken": "What fixed it",
    "confirmed_by": "What evidence shows it's fixed (metrics, logs)"
  },
  "data_gaps": ["List anything you cannot determine from the provided data"],
  "internal_details_to_exclude": ["Any internal names, infra details, or engineer names that must NOT appear in customer communications"]
}
```

## Rules
- Only state facts supported by the data. If you cannot determine something, put it in data_gaps.
- Distinguish confirmed root cause (engineers agreed on it) from initial hypotheses.
- Cross-reference timestamps across sources to build the timeline.
- All timestamps in UTC.
- Return ONLY valid JSON, no markdown fencing.

## User Message Construction
Concatenate all 5 files with clear delimiters:

```
=== PAGERDUTY INCIDENT ===
{contents of pagerduty_incident.json}

=== CLOUDWATCH LOGS ===
{contents of cloudwatch_logs.json}

=== PROMETHEUS METRICS ===
{contents of prometheus_metrics.json}

=== GITHUB DEPLOYMENTS ===
{contents of github_deployments.json}

=== INCIDENT CONTEXT (SLACK THREAD) ===
{contents of incident_context.txt}
```

## Expected Output for Validation
After getting the LLM response, parse the JSON and assert these key facts:

```python
expected = {
    "affected_service": "api-gateway",
    "severity": "SEV-2",
    "root_cause_contains": "HTTP client timeout",  # or "PR #12345" or "timeout from 10s to 30s"
    "trigger_time": "2025-01-15T14:15",  # deployment time, not alert time
    "resolution_contains": "rollback",
    "customer_impact_severity": "degraded_performance",  # NOT full_outage — most requests slow, some 500s
    "data_gaps_should_mention": ["regional scope", "which product features", "customer count"],
    "internal_details_should_include": ["rds-prod-main", "alice.engineer", "bob.sre", "john.dev"]
}
```

Print a pass/fail for each assertion. This is our ground truth test.

## Implementation Requirements
Keep it simple: One file, ~80 lines, no classes, no frameworks. Just read files → build prompt → call API → parse JSON → validate.

## Environment Setup
1. Copy `.env.example` to `.env`
2. Add your Anthropic API key to the `.env` file
