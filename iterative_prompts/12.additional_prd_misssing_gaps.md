## Group A — Described in the Spec, Not Found on the Results Page

**1. Zone 1 is entirely absent (Pre-Flight Context / Deployments Near Incident Onset)**
The spec describes Zone 1 as the first thing an IC sees: a deployment correlation banner flagging deployments as HIGH, LOW, or POST-ONSET relevance. The results page goes directly from the progress steps to "② STATUS PAGE DRAFT" — there is no Zone 1 section at all, and the zone numbering itself starts at ②, which confirms it's missing. This is particularly problematic because sample data *does* contain a qualifying deployment (PR #12345, api-gateway, 5 minutes before onset), which should have rendered a HIGH relevance flag. The spec explicitly says this zone is the most practically useful for an IC — surfacing the causal deployment without requiring log searches — and it's simply gone.

**2. The Quality Checks section is missing several of the specified high-severity checks**
The spec defines six high-severity checks and two distinct medium-severity checks. The results page shows only three issues: the "deployment_induced + no deployment flags" medium conflict, and "Missing: What's not affected" errors for Investigating and Monitoring. Critically absent:
- "False detection reassurance" — the spec says this catches comms that say detection is operational when it is actually degraded/offline. The Resolved message says "all systems are operating normally" while detection is Unknown — this seems like it should have triggered.
- "Missing detection confirmation when detection is fully_operational" — not shown (though here detection is Unknown, not operational, so this may be conditional).
- "portal_auth_failure + detection degraded" — this is a logic consistency check that appears nowhere on the page.
- "confirmed root cause + low confidence" check — root cause is confirmed + high confidence here so it may not apply, but there's no visible evidence this check is implemented at all.
- "deployment_induced + unknown root cause" — not shown.
- "third_party_dependency + no external trigger" — not shown.
The spec implies these checks should be invisible when they don't apply, but an IC has no way to know which checks are running vs. which ones were never implemented.

**3. The "Resolved" and "Identified" stage messages both omit Sentence 2 entirely (the mandatory "what's not affected" sentence), but the Quality Check flags are shown only for Investigating and Monitoring**
The spec says the mandatory second sentence must appear in *every* generated status update. The page flags the absence in Investigating and Monitoring but doesn't flag it for Identified or Resolved. The Resolved message ends on "all systems are operating normally" with no explicit statement about email detection or remediation. This inconsistency — flagging some stages but not others for the same missing element — suggests the validation isn't being applied uniformly.

**4. No source attribution shown for root cause, timeline, and customer impact claims (Zone 6)**
The spec describes the Raw Extraction Data zone as including "source attribution for root cause, timeline, and customer impact claims." The Structured Incident Analysis expander shows the timeline data, confidence scores, and data gaps, but there is no visible "source attribution" field linking specific claims to their source files.

**5. No customer impact classification field visible (Zone 6 / Zone 3)**
The spec describes a dedicated "Customer Impact Classification" field showing the severity assessment (degraded_performance / partial_outage / full_outage) as a top-level structured output. The word "partial_outage" never appears in the results display with that label — the IC Verification card says "How bad is it? — Partial outage..." in prose, but there's no formal structured field with the canonical classification value exposed.

## Group B — Present but Unclear, Confusing, or Missing Meaningful Detail

**1. The "Security Status Unconfirmed" badge renders as visible literal annotation text on screen**
Below the Resolved/duration line, the text "Right badge: security function impact (red = detection may be down)" is rendered as a sentence *directly on the page* rather than as a tooltip or helper text hidden in a secondary element. This reads as a developer comment accidentally surfaced in production UI. An IC or customer service manager seeing this would be confused — it reads as if the UI is narrating itself. The badge next to it ("? Security Status Unconfirmed") is actually the right approach, but the explanatory annotation below it should be a tooltip on hover or removed entirely, not static body text.

**2. The security function impact badge is orphaned from any contextual explanation of what it means for this incident**
The "? Security Status Unconfirmed" badge floats in the header area with no proximate explanation of *why* it's unconfirmed and what an IC should do about it. The actual reasoning ("api-gateway is a generic infrastructure service name without explicit evidence linking it to detection or remediation functions…") is buried in Zone 6, collapsed by default. An IC making a publish decision from Zones 1–3 would see "Security Status Unconfirmed" and have no guidance on how to resolve it before publishing.

**3. The word count badge coloring is explained by a legend, but the legend placement is confusing**
The green/orange/red word count legend appears *above* the message cards and reads "■ ≤100w ■ ≤110w ■ >110w — word count per message." But the color squares in the legend are very small and the text doesn't visually associate with the badge shown next to each stage label inside the cards (e.g. "61w" shown in green). An IC scanning quickly might not connect the legend to the per-message count, especially since the legend is separated from the cards by whitespace.

**4. The stale data warning is present but expressed in a confusing way**
The spec says Zone 4 should show a stale data warning if incident timestamps are more than 24 hours old. The page does show: "Analysis based on data from Jan 15, 2025 14:23 UTC (10449h ago). If the incident is still active, re-upload current data." The parenthetical "10449h ago" is technically accurate but bizarre in context — it's approximately 435 days of staleness expressed in raw hours, which reads more like a bug than a warning. For sample/test data this is expected, but the raw hour count rather than a human-friendly format ("over a year ago") would be genuinely alarming or confusing to a real IC in a live incident.

**5. The IC Verification zone shows a summary bar ("0/5 verified · 0 disputed · 5 unverified") but the copy lock state is not visually obvious until you scroll far down**
The spec describes the copy output as being "gated" — unlocked only when all 5 items are verified and zero are disputed. On the results page, the copy unlock state is only visible at the very bottom of Zone 4 in the "Ready to Publish?" checklist, which is a separate section well below the evidence cards. An IC who finishes verifying all cards might not realize there's still a separate checklist to complete before copy is enabled. The two gates (5 evidence cards + 5 checklist items) feel like one, but they are rendered in two physically separate locations with no visual connection between them.

**6. The "What's not broken?" evidence card is substantively weak, and the page doesn't call this out**
The IC Verification card for "What's not broken?" honestly states that the conclusion rests on *absence of evidence* (no errors appearing for notification-service) rather than positive confirmation. The page does flag this with a "💡 To verify" prompt, but this conclusion is labeled 🟡 medium confidence — which looks acceptable rather than genuinely concerning. For a security product where the IC must affirmatively confirm what's protected, absence-of-evidence for the "not broken" claim is the most operationally risky finding on the page, and it's displayed with the same visual treatment as stronger claims.

**7. The "Recent Incident Patterns" list in Zone 5 shows duplicate entries with near-identical descriptions**
The Analyst View shows four "Deployment Induced" entries with nearly identical one-line summaries, two of which are character-for-character duplicates. This looks like a logging/persistence bug, but from an IC's perspective it also destroys the value of the cross-incident pattern list — if you can't trust that the list represents distinct historical incidents, the pattern analysis loses credibility.

**8. The "Resolved" message drops the detection status sentence but doesn't flag this**
The Investigating, Identified, and Monitoring messages all contain "We are confirming the status of email security detection services…" (the correct `detection_status: unknown` sentence per spec). The Resolved message contains no equivalent — it simply says "all systems are operating normally." But detection was unknown, not confirmed operational. The spec states sentence 2 is mandatory and never repositioned or trimmed — yet the Resolved stage omits it with no warning flag from Quality Checks. This is both a Group A gap (the check isn't running) and a Group B issue (the message itself is arguably making implicit false reassurance by saying all systems are normal without explicitly addressing detection).

## Group C — Additional Observations (Bugs / Structural Issues)

**Zone 1's absence breaks the zone-number sequence on the page.** The results page shows "② STATUS PAGE DRAFT" and "③ QUALITY CHECKS" and "④ IC VERIFICATION" — but the ① is missing entirely. This means a user who is not familiar with the spec sees zone numbering that starts at ② with no explanation, which looks like either a UI error or a missing section. Even if Zone 1 is conditionally hidden (the spec says it should be hidden when no deployments are flagged), the sample data contains a directly relevant deployment that should have triggered it. Either the conditional logic is wrong, or the deployment correlation step is not running at all.

**The "Logic conflict" quality check (medium severity) is rendered in a yellow card above the red high-severity cards**, which inverts visual priority. The spec says high-severity checks should be shown inline in red and medium-severity in a collapsed expander. On screen, the yellow "Logic conflict" card appears *first* — above the red "Missing: What's not affected" cards. An IC scanning top-to-bottom would read the medium issue before the high-severity ones, which is the opposite of the intended decision order
