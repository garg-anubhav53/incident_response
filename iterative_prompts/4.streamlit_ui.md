# Prompt 4: Streamlit UI — Incident Analyzer & Status Page Generator

> **Status: Deployed.** This document reflects the actual implementation in `app.py`.

Build a single-file Streamlit app `app.py` that lets a user upload incident data files, runs the extraction + communication pipeline, and displays the result in a status-page-style format modeled after Abnormal AI's real status page.

## What the app does

1. **Upload panel**: User uploads any combination of the 5 supported files. A "Load Sample Data" button instantly pulls from `data/`. Not all files required.

2. **Deployment correlation** (pre-LLM): Before calling the extraction API, `correlate_deployments()` scans `github_deployments.json` and flags deployments within 60 minutes before incident onset (plus 30 minutes after, for post-onset worsening factors). Results are injected into the extraction prompt. Uses multi-metric Prometheus consensus to anchor onset time rather than a single anomalous reading.

3. **Extraction API call**: File contents sent to Claude with `EXTRACTION_SYSTEM_PROMPT`. Prometheus metrics are pre-processed into a two-part format: a human-readable anomaly summary plus verbatim timestamp-value pairs for anomalous series (enabling the LLM to cite specific metric values in `evidence_trace`). Response saved as structured dict + persisted to `inference_logs/`.

4. **Generation API call**: Extraction dict sent to Claude with `GENERATION_SYSTEM_PROMPT`. Response is the 4-stage communications dict. Stage order is enforced in Python after the LLM call regardless of LLM output order.

5. **Validation**: `validate_communications()` checks all stage messages for 5 keyword-based field checks (all stages including Investigating must have customer action). `validate_extraction_consistency()` runs cross-checks between `pattern_classification` and `security_function_impact` for logical contradictions.

6. **Results display**: 6 visually-zoned sections — Pre-Flight Context → Status Page Preview → Quality Checks → IC Verification → Analyst View → Raw Extraction Data.

## CSS Approach

All CSS injected via a single `st.markdown(CSS_STYLES, unsafe_allow_html=True)` block at module load.

**Critical rules learned in deployment:**
- Do NOT use broad `p, li, span, div { color !important }` — it overrides badge/stage-label colors
- Badge classes must set `color: white !important` directly on `.badge-resolved`, `.badge-degraded`, `.badge-outage`
- Stage label colors need `!important` on each rule
- `st.text()` produces `<pre>` with `white-space: pre` — use `st.markdown()` instead
- Newlines in LLM message output must be `.replace("\n", "<br>")` before HTML injection
- Deployment alerts use the custom `.deploy-high` CSS class (orange-bordered card) — NOT `st.warning()`

## Prompts

### EXTRACTION_SYSTEM_PROMPT
15 rules (Rule 0 through Rule 14). Key rules:
- **Rule 0**: Null over hallucination — unknown fields must be null/[], never invented
- **Rule 2**: ROLLBACK = CONFIRMATION — a successful rollback sets root_cause.status="confirmed"
- **Rule 11**: Pattern classification — 8 types + unknown; portal_auth_failure explicitly incompatible with detection degraded/offline
- **Rule 12**: Inference logging — every claim logged with inference_type + confidence
- **Rule 13**: Security function classification — full detection/remediation/other taxonomy
- **Rule 14**: Evidence trace — verbatim excerpts only, explicit what_is_not_broken absence guidance, specific verification suggestions

### GENERATION_SYSTEM_PROMPT
Key changes from earlier versions:
- "What's not affected" must be the SECOND sentence in every update (not buried at end)
- Customer action required in ALL stages including Investigating
- Rule 4 specifies trimming priority — never cut what's-not-affected to hit word count
- Identified stage guidance tied to root_cause.status (confirmed vs hypothesized vs unknown)
- Concrete example message provided as few-shot template
- DST handling specified (March–October = PDT = UTC-7)
