# Prompt 4: Streamlit UI — Incident Analyzer & Status Page Generator

> **Status: Deployed.** This document reflects the actual implementation in `app.py`.

Build a single-file Streamlit app `app.py` that lets a user upload incident data files, runs the extraction + communication pipeline, and displays the result in a status-page-style format modeled after Abnormal AI's real status page.

## What the app does

1. **Upload panel**: User uploads any combination of the 5 supported files. A "Load Sample Data" button instantly pulls from `data/`. Not all files required.

2. **Deployment correlation** (pre-LLM): Before calling the extraction API, a Python function scans `github_deployments.json` and flags deployments within 60 minutes before incident onset. Results are injected into the extraction prompt.

3. **Extraction API call**: File contents sent to Claude with `EXTRACTION_SYSTEM_PROMPT`. Prometheus metrics are pre-summarized (baseline/anomaly/recovery) before being included. Response saved as structured dict + persisted to `inference_logs/`.

4. **Generation API call**: Extraction dict sent to Claude with `GENERATION_SYSTEM_PROMPT`. Response is the 4-stage communications dict.

5. **Validation**: `validate_communications()` checks all 4 stage messages for 6 required fields and returns warnings.

6. **Results display**: Status page HTML → validation warnings → pattern analysis expander → structured analysis expander.

## CSS Approach

All CSS injected via a single `st.markdown(CSS_STYLES, unsafe_allow_html=True)` block at module load.

**Critical rules learned in deployment:**
- Do NOT use broad `p, li, span, div { color !important }` — it overrides badge/stage-label colors
- Do NOT use `h1-h6 { color !important }` — same issue
- Badge classes must set `color: white !important` directly on `.badge-resolved`, `.badge-degraded`, `.badge-outage`
- Stage label colors need `!important` on each rule (`.stage-resolved { color: #10b981 !important }`)
- `st.text()` produces `<pre>` with `white-space: pre` causing scrollable elements — use `st.markdown()` instead
- Newlines in LLM message output must be `.replace("\n", "<br>")` before HTML injection

## Session State Keys

| Key | Type | Description |
|-----|------|-------------|
| `step` | str | `"upload"` \| `"processing"` \| `"results"` |
| `extraction` | dict | Raw extraction response |
| `comms` | dict | Raw generation response |
| `uploaded_files` | dict | `{filename: content_str}` — read immediately on upload |
| `sample_data` | dict | Same shape, loaded from `data/` |
| `deployment_flags` | list | Output of `correlate_deployments()` |
| `validation_warnings` | list | Output of `validate_communications()` |
| `error` | str | Error message; shown as banner with Try Again button |

**Important**: `UploadedFile` objects must be read to strings immediately (`{f.name: f.read().decode("utf-8") for f in uploaded}`) before storing in session state — file cursors are exhausted after a Streamlit rerun.

## Prompts

### EXTRACTION_SYSTEM_PROMPT
Includes 12 rules. Key additions in v2 (Prompt 5):
- Rule 11: PATTERN CLASSIFICATION — 9 pattern types, definitions, pick strongest
- Rule 12: INFERENCE LOG — log every factual claim with claim/sources/inference_type/confidence

Output schema adds two new fields:
```json
"pattern_classification": {
  "primary_pattern": "deployment_induced | infrastructure_cascade | ...",
  "confidence": "high | medium | low",
  "reasoning": "2-3 sentences"
},
"inference_log": [
  {"claim": "...", "sources": [...], "inference_type": "direct_observation | cross_reference | absence_of_evidence | engineer_confirmation", "confidence": "high | medium | low"}
]
```

### GENERATION_SYSTEM_PROMPT
REQUIRED FIELDS specification replaces the old STRUCTURE PER UPDATE section. Every stage message must contain all 6 fields. The "What's not affected" field is flagged as CRITICAL and must never be omitted — it is the most important field for a security vendor.

## What NOT to build (still excluded)

- No database or persistence (inference_logs is the only disk write)
- No user auth
- No editing of extracted data
- No export/copy buttons
- No component health grid or uptime bars
- No historical incident list in the UI (recent patterns shown in expander only)
- No logo — text header only
